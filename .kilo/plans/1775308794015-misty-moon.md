## New Issues (Priority: Implement Now)

### Issue A: State Does Not Persist on Hard Refresh (Build Flow)

**Root Cause (Confirmed by Code Analysis)**

The build flow saves `{ foundationData, pillarData, currentPage }` to `sessionStorage` under key `okiru-processor-build-flow-v1`. On **hard refresh**, `DocumentProcessor.tsx` re-mounts but the restore only happens when `handleModeSelect('build')` is explicitly called (line 1342) — it is NOT called automatically on mount. So a hard refresh sends the user back to the mode-select (upload/manual) screen.

Additionally, the `flowMode` is `null` on first mount — the sessionStorage restore is gated behind clicking "Build Scorecard" again.

**Fix Required**

In `apps/web/src/pages/DocumentProcessor.tsx`:

1. **On component mount**, check if `sessionStorage.getItem('okiru-processor-build-flow-v1')` exists. If it does, AND `flowMode` is not yet set, auto-restore the build flow without waiting for the user to click "Build Scorecard" again.

2. **Implementation**: Add a `useEffect` with empty deps `[]` that:
   - Reads the sessionStorage key
   - If found and `currentPage` is `'build-pillars'` or `'build-foundation'`, calls `handleModeSelect('build')` automatically (which internally reads sessionStorage)
   - This restores the exact page and data state before the refresh

3. **What gets restored**: `foundationData`, `pillarData`, `currentPage` — already all saved to sessionStorage correctly

4. **What does NOT get restored** (acceptable losses):
   - `manifest` / `buildValues` — entity manifest data (re-fetched if needed)
   - `sessionId` — regenerated fresh (ephemeral identifier)

### Issue B: Toolkit (Scorecard View) State Not Saving When Navigating In

**Root Cause (Confirmed by Code Analysis)**

When clicking "Open in Toolkit" from the build scorecard view:
1. `calculateFromPillarData()` hydrates Zustand store with `build-<sessionId>` as `activeClientId`
2. The store is set in-memory only — **no API client is created**, no MongoDB record exists
3. On any navigation away and back, or hard refresh, `loadClientData('build-xxx')` is SKIPPED (line 66 of App.tsx) because `isBuildMode = true`
4. But the Zustand store is ephemeral — it resets on any full page reload
5. The Toolkit pillar pages (Ownership, Management, etc.) call store action methods (e.g. `addShareholder`) which fire `api.addShareholder(...)` — but the `clientId` is `'build-xxx'` which **does not exist in MongoDB**, so all API saves silently fail with 404

**The two distinct sub-problems:**

**B1: Build-mode clients are never persisted to the database**
- Fix: When `calculateFromPillarData` is called in DocumentProcessor, also call `POST /api/assessments/pillars` or `POST /api/clients` (via `bulk-import`) to create a real persistent client record in MongoDB with a real UUID clientId
- After creation, update `localStorage('okiru-pro-active-client')` with the REAL clientId (not `build-xxx`)
- The Toolkit will then load from API on refresh and all saves will work

**B2: The Toolkit Scorecard view has no explicit "Save" button — it relies on auto-save from store actions**
- This is correct by design — every `addEmployee`, `addShareholder` etc. fires an API PUT
- BUT this only works if the client exists in DB (see B1)
- Once B1 is fixed, auto-save will work correctly

**Implementation Plan for B1:**

In `apps/web/src/pages/DocumentProcessor.tsx`, `calculateFromPillarData()` (lines 1456–1671):

1. After the local Zustand hydration (line 1487), call `POST /api/assessments` or `POST /api/clients/:id/bulk-import` with all the pillar data to create a real DB record
2. The `POST /api/assessments` endpoint (server/routes.ts line 1929) already does exactly this — it creates a client + all pillar data in one shot
3. Map `pillarData` + `foundationData` to the assessment payload format
4. On success, use the returned `clientId` (real UUID) instead of `build-xxx`
5. Update `localStorage.setItem('okiru-pro-active-client', realClientId)`
6. Update the Zustand store: `useBbeeStore.setState({ activeClientId: realClientId, client: { ...client, id: realClientId } })`
7. Navigate to `/toolkit/${realClientId}`

This is a significant fix that ensures all scorecard data is actually persisted.

### Issue C: AutoFill Button Hidden in Production

**Root Cause**

`AutoFillButton.tsx` line 56: `if (process.env.NODE_ENV === 'production') return null;`

The app is deployed in production mode, so the button never renders.

**Fix Required**

Change the production guard from `process.env.NODE_ENV === 'production'` to an explicit opt-in environment variable:

**Option 1 (Recommended): Use a separate env var flag**
```tsx
// Before: if (process.env.NODE_ENV === 'production') return null;
// After:
if (import.meta.env.PROD && !import.meta.env.VITE_ENABLE_AUTOFILL) return null;
```

Set `VITE_ENABLE_AUTOFILL=true` in the Kubernetes ConfigMap for the web deployment. This allows autofill in production while keeping it opt-in.

**Decision: Admin users only.**

Replace the `process.env.NODE_ENV === 'production'` guard with a role check:
```tsx
const { user } = useAuth(); // from existing auth context
if (user?.role !== 'admin') return null;
```

**Files to change:**
- `apps/web/src/components/AutoFillButton.tsx` — replace line 56 with role check
- Import `useAuth` from the existing auth context (already used elsewhere in the app)

### Issue D: Sector Template → Scorecard Link is Incomplete

**Root Cause**

When a user selects sector (e.g., `ICT`) in `ClientInformationForm.tsx`, the sector is stored in `foundationData.sectorCode`. It flows to `store.client.sectorCode` during hydration.

**The gap**: The Toolkit's `calculatorConfig` (which drives which sector-specific weights and thresholds are applied) is **never loaded in build-mode**. In `calculateFromPillarData`, `calculatorConfig` is set to `null` (line 1489). The Toolkit calculators use the default RCOGP weights regardless of sector.

**Important Clarifications:**
- Sector selection is **already compulsory** in `ClientInformationForm.tsx` - this is working correctly
- The issue is that `calculatorConfig` is never populated from the selected sector in build mode
- **Primary source**: Calculations must come from ArangoDB (seeded flows) - see Phase 3 in `bbee-scoring-engine-overhaul.md`
- **Fallback**: Hardcoded logic in `sectorConfig.ts` only if ArangoDB query fails

**Fix Required**

In `calculateFromPillarData()`:
1. **Query ArangoDB first**: Call the calculation engine API to get sector-specific rules from `sector_rules` collection
2. If ArangoDB returns sector config, use it to populate `calculatorConfig`
3. **Fallback only on failure**: Use `getSectorConfig(sectorCode, scorecardType)` from `sectorConfig.ts`
4. Update `useBbeeStore.setState({ calculatorConfig: sectorConfig })`

**Files to change:**
- `apps/web/src/pages/DocumentProcessor.tsx` — `calculateFromPillarData()`: Add ArangoDB API call to fetch sector rules
- `apps/web/Toolkit/src/lib/store.ts` — Ensure `calculatorConfig` state is properly typed and used by calculators
- `apps/api/pipeline/rulesRegistry.ts` — Ensure rules are exposed via API endpoint

## Issue 7: Session/Authentication Bug – "Open in Toolkit" Logs User Out

### Root Cause Analysis
The flow when clicking "Open in Toolkit":
1. DocumentProcessor calls `navigate('/toolkit/${buildClientId}')` (wouter client-side navigation)
2. `ToolkitView.tsx` stores `clientId` in `localStorage` and renders `ToolkitApp`
3. `ToolkitApp` wraps with `AuthProvider` which calls `GET /api/auth/me` on mount
4. **Both** `apps/web/src/App.tsx` (outer) and `apps/web/Toolkit/src/App.tsx` (inner) have `AuthProvider`
5. When `ToolkitView` renders the inner `ToolkitApp`, a **second** `AuthProvider` is mounted
6. The inner `AuthProvider` fires another `fetch /api/auth/me` — if this races or the session cookie isn't sent correctly, it fails and `setUser(null)` is called
7. `AuthenticatedApp` in the inner Toolkit app then shows `LandingPage` or redirects to `/auth`
8. Additionally, the `ProtectedRoute` in the outer app (`RouteGuards.tsx`) wraps the toolkit route — if the user state flickers to null momentarily (during re-mount), it navigates to `/auth`

### Required Fix

#### A. `apps/web/Toolkit/src/App.tsx`
- The Toolkit `App` component re-wraps with `AuthProvider` even when embedded. When rendered inside `ToolkitView` (i.e., inside the outer app's `AuthProvider`), this creates a double-wrap.
- **Solution**: Export a `ToolkitEmbedded` variant (or `AppRoutes` component, which already exists) that skips `AuthProvider` wrapping, and use it in `ToolkitView.tsx`

#### B. `apps/web/src/pages/ToolkitView.tsx`
- Instead of rendering `<ToolkitApp />` (which creates new `AuthProvider`), render `<ToolkitEmbedded />` (or the existing `AppRoutes` component wrapped in just the needed providers, excluding `AuthProvider`)

#### C. Investigate Session Cookie
- The inner `AuthProvider` `fetch` uses `credentials: 'include'` — verify CORS and cookie `sameSite`/`domain` settings are correct
- Check `apps/api/src/index.ts` or main server file for session cookie configuration

#### D. Ensure Re-authentication Works
- If session expires while in toolkit, the error boundary in `DataLoader` already shows a "Session expired — Log in again" screen with a `navigate('/')` button
- Verify this works correctly end-to-end and doesn't get stuck in a redirect loop

## Key Clarifications / Decisions

1. **Issue 1 (EE levels)**: Per OWNER FEEDBACK, **Skilled Technical, Semi-skilled, and Unskilled** must ALL be added as Employment Equity designation levels. EAP target mapping: Skilled Technical → Uses **Middle** EAP targets; Semi-skilled → Uses **Junior** EAP targets; Unskilled → Uses **Junior** EAP targets.

2. **Issue 7 (Auth bug)**: The primary fix is to avoid double-wrapping `AuthProvider`. The existing `AppRoutes` component in `Toolkit/src/App.tsx` (line 123) can be used directly from `ToolkitView` after wrapping with the required non-auth providers (ThemeProvider, QueryClientProvider, TooltipProvider).

3. **Issue 5 (Graduation validation)**: This is implemented as a **warning**, not a hard block — the scoring still awards the point, but a yellow/amber notice appears if no ED beneficiary name matches any supplier in the procurement list.

4. **Issue 3 (Foreign Supplier)**: Based on B-BBEE Amended Codes, foreign suppliers' spend does not qualify for **Empowering Supplier** recognition (the 80% TMPS empowering supplier indicator). The flag is captured per supplier. In the calculator, flagged foreign suppliers will be excluded from the Empowering Supplier spend calculation but their spend still counts toward TMPS (unless the practitioner excludes them via TMPS calculation). This matches the B-BBEE toolkit's "Discretionary Exclusion" approach.

5. **Issue 6 (Category F)**: The TOOLKIT_TAB_MAP.md Section 5.1 shows Category F has a 15% cap and is for "External Unaccredited Training". The current code incorrectly labels it "YES Initiative" in `SkillsForm.tsx` and "Other Costs" in `SkillsDevelopment.tsx` — both need correction.
