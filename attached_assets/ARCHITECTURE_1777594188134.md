## 1. The problem this solves

Without this library, database code tends to look like this:

```typescript
// Route handler — Express, before migration
router.post('/users', async (req, res) => {
  const result = await pool.query(
    'INSERT INTO users (name, email) VALUES ($1, $2) RETURNING *',
    [req.body.name, req.body.email],
  )
  res.status(201).json(result.rows[0])
})
```

This has several problems:

- **SQL is scattered.** Every file that touches users has its own SQL string.
  When the schema changes, you hunt through every file.
- **No transaction safety.** If two queries in the same request need to be
  atomic, there is no single place that handles COMMIT and ROLLBACK.
- **Hard to test.** You cannot run this handler in a unit test without a real
  Postgres connection.
- **Impossible to swap databases.** `pool.query(...)` is Postgres-specific.
  Moving to MongoDB means rewriting every file.

This library solves all four problems with three patterns.

## 2. The three patterns

### 2.1 Repository

A **Repository** is an object that knows how to read and write one type of
domain entity. It hides all database-specific code behind a plain interface.

```typescript
// TypeScript — core interface (typescript/src/abstractions/repository.ts)
export interface IRepository<TEntity, TId> {
  findById(id: TId): Promise<TEntity | null>
}
```

```python
# Python — core interface (python/src/data_layer/abstractions/repository.py)
class IRepository(Protocol, Generic[T, TId]):
    async def find_by_id(self, entity_id: TId) -> Optional[T]: ...
```

The base interface only requires `findById` / `find_by_id`. You extend it per
entity to add operations specific to that entity:

```typescript
// In your service or a shared domain package
export interface IUserRepository extends IRepository<User, number> {
  findByEmail(email: string): Promise<User | null>
  save(data: { name: string; email: string }): Promise<User>
}
```

The concrete implementation of `IUserRepository` for Postgres lives in
`typescript-postgres/src/repositories/postgres-user-repository.ts` and uses
Drizzle ORM. The concrete implementation for MongoDB lives in
`typescript-mongo/src/repositories/mongo-user-repository.ts` and uses the
MongoDB Node.js driver. The route handler imports only `IUserRepository` — it
never touches either concrete class.

**Rule:** A repository never calls `commit()` or `rollback()`. It only runs
queries. Transaction control belongs exclusively to the Unit of Work.

---

### 2.2 Unit of Work

A **Unit of Work** (UoW) represents a single transactional boundary. One
request = one UoW = one open database transaction.

```typescript
// TypeScript — core interface (typescript/src/abstractions/unit-of-work.ts)
export interface IUnitOfWork {
  commit(): Promise<void>
  rollback(): Promise<void>
}
```

```python
# Python — core interface (python/src/data_layer/abstractions/unit_of_work.py)
class IUnitOfWork(ABC):
    @abstractmethod
    async def commit(self) -> None: ...

    @abstractmethod
    async def rollback(self) -> None: ...
```

The concrete UoW (e.g. `PostgresUnitOfWork`) does three things:
1. Checks out a connection from the pool and sends `BEGIN` when created.
2. Exposes domain repositories (e.g. `.users`) as properties. Every
   repository on the UoW shares the same connection, so all queries run
   inside the same transaction automatically.
3. On `commit()`: sends `COMMIT` and releases the connection back to the pool.
   On `rollback()`: sends `ROLLBACK` and releases the connection.

```typescript
// Concrete UoW (typescript-postgres/src/postgres-unit-of-work.ts)
export interface IPostgresUnitOfWork extends IUnitOfWork {
  readonly users: IUserRepository   // <-- domain repositories live here
}

export class PostgresUnitOfWork implements IPostgresUnitOfWork {
  readonly users: IUserRepository

  private constructor(private readonly client: PoolClient) {
    const db = drizzle(client)          // same client = same transaction
    this.users = new PostgresUserRepository(db)
  }

  static async begin(pool: Pool): Promise<PostgresUnitOfWork> {
    const client = await pool.connect() // check out from pool
    await client.query('BEGIN')         // open the transaction
    return new PostgresUnitOfWork(client)
  }

  async commit(): Promise<void> {
    try {
      await this.client.query('COMMIT')
    } finally {
      this.client.release()             // always return to pool
    }
  }
  // ... rollback() mirrors commit()
}
```

A route handler that needs two operations on different repositories (e.g.
create a user AND create an order) calls both repositories on the same UoW,
then calls `commit()` once. Either both writes land, or neither does.

### 2.3 Data Access Factory

A **Data Access Factory** is the object that creates UoW instances. It holds
the long-lived connection pool (one pool per process) and produces a fresh UoW
for each request.

```typescript
// TypeScript — core interface (typescript/src/abstractions/data-access-factory.ts)
export interface IDataAccessFactory {
  createUnitOfWork(): Promise<IUnitOfWork>
}
```

```python
# Python — core interface (python/src/data_layer/abstractions/data_access_factory.py)
class IDataAccessFactory(ABC):
    @abstractmethod
    async def create_unit_of_work(self) -> IUnitOfWork: ...
```

The concrete factory (e.g. `PostgresDataAccessFactory`) holds a `Pool` (pg) or
`AsyncEngine` (SQLAlchemy) that is created once at startup and reused for
every request. Calling `createUnitOfWork()` / `create_unit_of_work()` checks
out one connection from that pool.

```typescript
// Concrete factory (typescript-postgres/src/postgres-data-access-factory.ts)
export class PostgresDataAccessFactory implements IDataAccessFactory {
  private readonly pool: Pool

  constructor(config: PostgresConfig) {
    this.pool = new Pool({             // created ONCE at startup
      connectionString: config.connectionString,
      max: config.maxConnections ?? 10,
    })
  }

  async createUnitOfWork(): Promise<IUnitOfWork> {
    return PostgresUnitOfWork.begin(this.pool)  // one per request
  }
}
```

**Critical rule:** Create the factory **once** at startup. Never create it
inside a request handler. Creating a `Pool` or `AsyncEngine` per request
causes connection exhaustion very quickly.

## 4. Startup — the composition root

The **composition root** is the one place in the entire application that
imports a concrete provider package. Everything else (routes, middleware,
services) imports interfaces only.

### TypeScript — `example-express/src/server.ts`

```typescript
import 'dotenv/config'          // 1. load .env into process.env
import { loadFactory } from './provider'
import { createApp } from './app'

const factory = await loadFactory()  // 2. read DATA_PROVIDER, build factory
const app = createApp(factory)       // 3. wire middleware + routes (interface only)
app.listen(3000)
```

`loadFactory()` (`provider.ts`) is the only file that does:

```typescript
import { PostgresDataAccessFactory, loadPostgresConfig }
  from '@company/data-layer-postgres'
```

Every other file in the app imports only:

```typescript
import type { IDataAccessFactory, IUnitOfWork } from '@company/data-layer'
```

### Python — `example-fastapi/src/example_fastapi/main.py`

```python
@asynccontextmanager
async def lifespan(app: FastAPI):
    app.state.factory = await load_factory()  # created ONCE at startup
    yield                                     # app runs here
    # shutdown cleanup would go here
```

`load_factory()` (`provider.py`) is the only file that imports a concrete
provider class. All routers import only `IUnitOfWork` from the core package.

## 5. Request lifecycle

### 5.1 Express (TypeScript)

Every incoming HTTP request passes through this sequence:

```
HTTP Request
    │
    ▼
attachUow middleware
    │  factory.createUnitOfWork()
    │  ── calls PostgresUnitOfWork.begin(pool)
    │  ── pool.connect()  → checks out a PoolClient
    │  ── client.query('BEGIN')
    │  ── req.uow = new PostgresUnitOfWork(client)
    ▼
Route handler (e.g. POST /users)
    │  const user = await req.uow.users.save({ name, email })
    │  ── runs INSERT inside the open transaction
    │  await req.uow.commit()
    │  ── client.query('COMMIT')
    │  ── client.release()  → returns connection to pool
    ▼
HTTP Response (201 Created)
```

On error:

```
Route handler throws
    │
    ▼
next(err) — passes to global error handler (app.ts)
    │  await req.uow.rollback()
    │  ── client.query('ROLLBACK')
    │  ── client.release()
    ▼
HTTP Response (500 Internal Server Error)
```

Key points:
- The connection is **always** released — in `finally` blocks inside
  `commit()` and `rollback()`. There are no connection leaks.
- The route handler never imports `pg` or Drizzle. It only uses `req.uow`.
- `attachUow` is registered with `app.use()` before the routes, so every
  request has a UoW regardless of which route is matched.

### 5.2 FastAPI (Python)

FastAPI uses dependency injection (`Depends`) instead of middleware.

```
HTTP Request
    │
    ▼
get_factory dependency
    │  returns request.app.state.factory (set at startup)
    ▼
get_uow dependency
    │  uow = await factory.create_unit_of_work()
    │  ── AsyncSession(engine) → BEGIN
    ▼
Route handler (e.g. POST /users)
    │  user = await uow.users.save(name=body.name, email=body.email)
    │  (no explicit commit here — get_uow handles it)
    ▼
get_uow dependency (resumes after yield)
    │  await uow.commit()
    │  ── session.commit() → COMMIT
    │  ── session.close()  → returns connection to pool
    ▼
HTTP Response (201 Created)
```

On error (exception raised in route handler):

```
Route handler raises HTTPException or any other exception
    │
    ▼
get_uow dependency (except block)
    │  await uow.rollback()
    │  ── session.rollback() → ROLLBACK
    │  ── session.close()
    │  raise    ← re-raises so FastAPI still returns the error response
    ▼
HTTP Response (404 / 409 / 500)
```

The `yield` pattern in `get_uow` is the Python/FastAPI equivalent of
Express's middleware + error handler combined. Route handlers never call
`commit()` or `rollback()` themselves.

## 6. Concrete implementations

### 6.1 TypeScript + PostgreSQL

**Connection pool:** `pg.Pool` — one pool per process, shared across all
requests.

**ORM:** Drizzle ORM — used inside `PostgresUserRepository`. Drizzle receives
the `PoolClient` that already has `BEGIN` open, so every Drizzle query runs
inside the same transaction automatically.

**Schema:** `typescript-postgres/src/schema/users.ts` defines the Drizzle
table schema. Migrations are generated with `npm run db:generate` and applied
with `npm run db:migrate` (drizzle-kit).

**Transaction lifecycle inside UoW:**

```typescript
// begin() — called by the factory
static async begin(pool: Pool): Promise<PostgresUnitOfWork> {
  const client = await pool.connect()   // checkout
  await client.query('BEGIN')
  return new PostgresUnitOfWork(client)
}

// commit() / rollback() — both use finally to guarantee release
async commit(): Promise<void> {
  try {
    await this.client.query('COMMIT')
  } finally {
    this.client.release()   // always runs, even if COMMIT throws
  }
}
```

**Config:** `loadPostgresConfig()` reads `DATABASE_URL` and `DB_POOL_SIZE`
from environment variables. It throws a clear error at startup if
`DATABASE_URL` is missing, so the app never starts with a broken config.

**Connection string format:**
```
postgresql://user:password@host:port/database
```

---

### 6.2 TypeScript + MongoDB

**Client:** `mongodb.MongoClient` — one client per process. The client
manages its own internal connection pool.

**Transactions:** MongoDB transactions require a replica set. A standalone
MongoDB does not support them. In development, use:

```bash
docker run -p 27017:27017 -d mongo:7 --replSet rs0
docker exec -it mongo mongosh --eval "rs.initiate()"
```

**Session lifecycle inside UoW:**

```typescript
static async begin(client: MongoClient, dbName: string): Promise<MongoUnitOfWork> {
  const session = client.startSession()
  session.startTransaction()  // synchronous — marks the session as "in txn"
  const db = client.db(dbName)
  return new MongoUnitOfWork(session, db)
}

async commit(): Promise<void> {
  try {
    await this.session.commitTransaction()
  } finally {
    await this.session.endSession()   // always returns session to driver pool
  }
}
```

Note: `startTransaction()` is synchronous in the Node.js driver. The actual
network round-trip happens on the first query, not at `begin()`.

**Extra startup step:** Unlike Postgres, you must call `factory.connect()`
once at startup to verify connectivity:

```typescript
const factory = new MongoDataAccessFactory(config)
await factory.connect()   // pings MongoDB; fails fast if unreachable
```

**Connection string format:**
```
mongodb://user:password@host:port          (self-hosted)
mongodb+srv://user:password@cluster.net    (Atlas / cloud)
```

`MONGO_DATABASE` is set separately because the connection string does not
include the database name for MongoDB.

---

### 6.3 Python + PostgreSQL

**Engine:** SQLAlchemy 2 `AsyncEngine` — one engine per process, wraps the
asyncpg connection pool.

**Session:** `AsyncSession` — one session per request, scoped to a single
transaction. The session receives `expire_on_commit=False` because the session
is closed immediately after commit; re-fetching expired objects would fail.

**ORM:** SQLAlchemy 2 ORM — `python-postgres/src/data_layer_postgres/schema/models.py`
defines the mapped models. The repository uses `session.get()` and
`session.execute(select(...))`.

**Transaction lifecycle:**

```python
@classmethod
async def begin(cls, engine: AsyncEngine) -> "PostgresUnitOfWork":
    session = AsyncSession(engine, expire_on_commit=False)
    await session.begin()   # sends BEGIN
    return cls(session)

async def commit(self) -> None:
    try:
        await self._session.commit()
    finally:
        await self._session.close()   # always releases the connection
```

**Connection string format (Python only):**
```
postgresql+asyncpg://user:password@host:port/database
```

The `+asyncpg` driver suffix is **required** for SQLAlchemy async. The
`provider.py` helper `_ensure_asyncpg()` automatically corrects
`postgresql://` or `postgres://` URLs so the app gives a clear error message
instead of a cryptic SQLAlchemy exception.

---

### 6.4 Python + MongoDB

**Client:** `motor.motor_asyncio.AsyncIOMotorClient` — motor is MongoDB's
async Python driver. One client per process.

**Transactions:** Same replica set requirement as the TypeScript MongoDB
provider. See section 6.2 for Docker setup.

**Session lifecycle:** Mirrors the TypeScript MongoDB provider — uses
`start_session()` and `start_transaction()` / `commit_transaction()` /
`abort_transaction()`.

## 7. Provider selection

The `InMemoryProviderRegistry` (in both core packages) maps a string key to
a factory. At startup the composition root registers each factory:

```typescript
// TypeScript (InMemoryProviderRegistry)
const registry = new InMemoryProviderRegistry()
registry.registerFactory('postgres', postgresFactory)
registry.registerFactory('mongodb', mongoFactory)

const factory = registry.getFactory(process.env.DATA_PROVIDER ?? 'postgres')
```

The example apps use a simpler `loadFactory()` function (in `provider.ts` /
`provider.py`) that does the same thing with a direct `if`/`else`:

```typescript
export async function loadFactory(): Promise<IDataAccessFactory> {
  const provider = process.env.DATA_PROVIDER ?? 'postgres'

  if (provider === 'postgres') {
    const { PostgresDataAccessFactory, loadPostgresConfig } =
      await import('@company/data-layer-postgres')
    return new PostgresDataAccessFactory(loadPostgresConfig())
  }

  if (provider === 'mongodb') {
    const { MongoDataAccessFactory, loadMongoConfig } =
      await import('@company/data-layer-mongo')
    const factory = new MongoDataAccessFactory(loadMongoConfig())
    await factory.connect()
    return factory
  }

  throw new Error(`Unknown DATA_PROVIDER: "${provider}"`)
}
```

Dynamic `import()` is used so neither provider package is loaded unless it is
actually selected. If only `@company/data-layer-postgres` is installed,
setting `DATA_PROVIDER=mongodb` throws an informative error at startup rather
than crashing on the first request.

## 8. Observability (OpenTelemetry)

All four concrete UoW implementations instrument `commit()` and `rollback()`
with OpenTelemetry spans. No configuration is required in the UoW code — if
no OTel SDK is configured in the application, all spans are silent no-ops
(zero overhead, zero crashes).

```typescript
// In commit() and rollback() of every concrete UoW
const span = tracer.startSpan('unit_of_work.commit')
span.setAttribute('db.provider', 'postgres')  // or 'mongodb'
try {
  await this.client.query('COMMIT')
  span.setStatus({ code: SpanStatusCode.OK })
} catch (err) {
  const { traceId } = span.spanContext()
  // Log the traceId — lets you find this exact failure in your trace backend
  console.error(`[data-layer-postgres] commit failed  traceId=${traceId}`, err)
  span.recordException(err)
  span.setStatus({ code: SpanStatusCode.ERROR, message: String(err) })
  throw err
} finally {
  this.client.release()
  span.end()
}
```

When an OTel SDK is configured (e.g. with Jaeger or Honeycomb), every
commit and rollback appears as a span in the distributed trace with:
- `db.provider` = `"postgres"` or `"mongodb"`
- Status: `OK` or `ERROR`
- On failure: the exception recorded on the span and the trace ID logged to
  stderr for cross-referencing

To enable OTel in an application, configure an SDK before calling
`loadFactory()`. The data layer packages need no changes.

## 9. Testing strategy

The library is designed so that service logic can be tested without any
database, network, or environment variables.

### The fake classes

**`FakeUnitOfWork`** (both TS and Python) implements `IUnitOfWork` with no
side effects. It tracks state in plain fields:

```typescript
class FakeUnitOfWork implements IUnitOfWork {
  committed = false
  rolledBack = false
  commitCount = 0
  shouldFailOnCommit = false   // set to true to simulate a DB failure

  async commit(): Promise<void> {
    if (this.shouldFailOnCommit) throw new Error('simulated commit failure')
    this.committed = true
    this.commitCount++
  }

  async rollback(): Promise<void> {
    this.rolledBack = true
  }
}
```

**`FakeDataAccessFactory`** wraps a `FakeUnitOfWork` and returns it from
`createUnitOfWork()`:

```typescript
class FakeDataAccessFactory implements IDataAccessFactory {
  constructor(private readonly uow: IUnitOfWork = new FakeUnitOfWork()) {}

  async createUnitOfWork(): Promise<IUnitOfWork> {
    return this.uow
  }
}
```

### The testing pattern

Follow these four steps in every unit test:

**Step 1** — Build a fake repository (in-memory, no SQL):

```typescript
class FakeUserRepository implements IUserRepository {
  private store = new Map<number, User>()
  private nextId = 1

  async findById(id: number) { return this.store.get(id) ?? null }
  async findByEmail(email: string) {
    return [...this.store.values()].find(u => u.email === email) ?? null
  }
  async save(data: { name: string; email: string }) {
    const user = { id: this.nextId++, ...data }
    this.store.set(user.id, user)
    return user
  }
}
```

**Step 2** — Extend `FakeUnitOfWork` with the repositories the service needs:

```typescript
class FakePostgresUow extends FakeUnitOfWork implements IPostgresUnitOfWork {
  constructor(readonly users: IUserRepository) { super() }
}
```

**Step 3** — Wire the fake factory and run the service:

```typescript
const fakeRepo = new FakeUserRepository()
const fakeUow  = new FakePostgresUow(fakeRepo)
const factory  = new FakeDataAccessFactory(fakeUow)

const user = await createUser(factory, 'Alice', 'alice@example.com')
```

**Step 4** — Assert on both the service output and the transaction state:

```typescript
assert.equal(user.name, 'Alice')
assert.equal(fakeUow.committed,  true)   // service called commit()
assert.equal(fakeUow.rolledBack, false)  // service did NOT roll back
```

### Testing the error path

Set `shouldFailOnCommit = true` before running the service:

```typescript
fakeUow.shouldFailOnCommit = true

await assert.rejects(
  () => createUser(factory, 'Bob', 'bob@example.com'),
  /simulated commit failure/,
)

assert.equal(fakeUow.committed,  false)
assert.equal(fakeUow.rolledBack, true)   // service caught the error and rolled back
```

This pattern gives you full branch coverage (happy path + error path) with
no database, no Docker, and no network — tests run in milliseconds.

See the working examples:
- TypeScript: `typescript-postgres/scripts/test-unit.ts`
- Python: `python-postgres/scripts/test_unit.py`

## 10. How to add a new entity

This walkthrough adds an `Order` entity. Steps are the same for TypeScript
and Python — only the syntax differs.

### Step 1 — Define the domain type and repository interface (in your service or shared domain package)

```typescript
// src/domain/order.ts
import type { IRepository } from '@company/data-layer'

export interface Order {
  id: number
  userId: number
  total: number
}

export interface IOrderRepository extends IRepository<Order, number> {
  findByUserId(userId: number): Promise<Order[]>
  save(data: { userId: number; total: number }): Promise<Order>
}
```

### Step 2 — Add the schema (TypeScript + Postgres)

```typescript
// typescript-postgres/src/schema/orders.ts
import { integer, pgTable, serial } from 'drizzle-orm/pg-core'

export const orders = pgTable('orders', {
  id:     serial('id').primaryKey(),
  userId: integer('user_id').notNull(),
  total:  integer('total').notNull(),
})
```

Generate and apply the migration:

```bash
npm run db:generate
npm run db:migrate
```

### Step 3 — Write the concrete repository

```typescript
// typescript-postgres/src/repositories/postgres-order-repository.ts
import { eq } from 'drizzle-orm'
import type { NodePgDatabase } from 'drizzle-orm/node-postgres'
import type { Order, IOrderRepository } from '../domain/order'
import { orders } from '../schema/orders'

export class PostgresOrderRepository implements IOrderRepository {
  constructor(private readonly db: NodePgDatabase) {}

  async findById(id: number): Promise<Order | null> {
    const [row] = await this.db.select().from(orders).where(eq(orders.id, id))
    return row ?? null
  }

  async findByUserId(userId: number): Promise<Order[]> {
    return this.db.select().from(orders).where(eq(orders.userId, userId))
  }

  async save(data: { userId: number; total: number }): Promise<Order> {
    const [row] = await this.db.insert(orders).values(data).returning()
    if (!row) throw new Error('INSERT into orders returned no rows')
    return row
  }
}
```

## 12. Design decisions and tradeoffs

### Why interfaces in a separate core package?

If interfaces lived inside a provider package (e.g.
`@company/data-layer-postgres`), then application code would have to depend
on the postgres package even when it does not use Postgres. Separating
interfaces into `@company/data-layer` (zero dependencies) keeps the
dependency graph clean.

### Why not use an ORM directly?

ORMs (Drizzle, SQLAlchemy, Prisma) are used *inside* the provider packages,
not *in* application code. Application code only ever calls
`uow.users.save(...)`. This means:

- You can swap the ORM inside a provider without touching application code.
- You can mock the repository in unit tests without loading the ORM at all.
- Different providers can use different ORMs (Drizzle for TS, SQLAlchemy for
  Python) without any inconsistency visible to the application.

### Why Unit of Work instead of just injecting repositories?

Without a UoW, if you inject two repositories and both need to run in the same
transaction, you have no clean way to share a connection between them. The UoW
holds the connection and exposes all repositories as properties — they all
share the same connection automatically.

### Why one UoW per request?

This keeps the transactional scope clear and predictable. One request does one
unit of work: either everything in the request succeeds (commit) or nothing
does (rollback). This prevents partial writes that leave data in an
inconsistent state.

Long-running background tasks may need a different scoping strategy (e.g. one
UoW per chunk), but for HTTP request handlers one UoW per request is the
correct default.

### Why dynamic import in provider.ts?

Using `await import('@company/data-layer-postgres')` means the postgres
package is only loaded if `DATA_PROVIDER=postgres`. If only one provider is
installed and the wrong `DATA_PROVIDER` is set, the error is immediate and
informative rather than a module-not-found crash somewhere deep in startup.

### Why no `IRepository.save()` on the base interface?

Different entities have different write shapes. A user save needs `name` and
`email`; an order save needs `userId` and `total`. Putting a generic
`save(data: unknown)` on the base would be un-typed and misleading. Entity-
specific write methods belong on the entity-specific interface extension.

### Why FakeUnitOfWork instead of mocking frameworks?

Mocking frameworks (sinon, unittest.mock) create mocks at runtime that can
drift from the real interface. A hand-written `FakeUnitOfWork` is:
- Typed: TypeScript will fail to compile if the fake does not match the
  interface.
- Explicit: its behaviour is immediately visible in the test file.
- Portable: no test framework dependency.

The `shouldFailOnCommit` flag lets you cover the error path without any
patching.
