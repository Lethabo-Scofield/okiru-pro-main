# Internal Publishing & Versioning Guide

This document covers how to version, release, and publish the data layer
packages to internal registries so services can pin to stable releases.

---

## Packages

| Package | Language | Registry |
|---------|----------|----------|
| `@company/data-layer` | TypeScript | Internal npm |
| `@company/data-layer-postgres` | TypeScript | Internal npm |
| `@company/data-layer-mongo` | TypeScript | Internal npm |
| `company-data-layer` | Python | Internal PyPI |
| `company-data-layer-postgres` | Python | Internal PyPI |
| `company-data-layer-mongo` | Python | Internal PyPI |

## Semantic Versioning Rules

All packages follow [semver](https://semver.org): `MAJOR.MINOR.PATCH`

### MAJOR — breaking change (consumers must update their code)

Bump MAJOR when you change a public interface in a way that breaks existing
implementations.

Examples:
- Adding a required method to `IUnitOfWork`, `IRepository`, or `IDataAccessFactory`
- Changing the signature of `commit()`, `rollback()`, or `createUnitOfWork()`
- Renaming or removing an exported type or class
- Changing a method parameter from optional to required

> **Rule:** If a consumer's code compiles today but would fail after the
> upgrade without changes, it is a MAJOR bump.

### MINOR — new capability, fully backwards compatible

Bump MINOR when you add something new that existing consumers can ignore.

Examples:
- Adding an optional method to an interface
- Adding a new provider package (`data-layer-redis`, etc.)
- Adding a new helper or utility export
- Adding a new optional config field

### PATCH — bug fix or internal improvement

Bump PATCH for fixes that do not change the public interface at all.

Examples:
- Fixing a connection leak in a UoW
- Correcting OTel span attribute names
- Updating a dependency to patch a CVE
- Fixing a config validation message

## Core vs Provider versioning

The core packages (`@company/data-layer`, `company-data-layer`) define the
interfaces. Provider packages depend on them.

**Rule:** Core packages should be versioned conservatively. A MAJOR bump to
core forces a MAJOR bump in every provider package.

| Change | Core | Providers |
|--------|------|-----------|
| New required method on IUnitOfWork | MAJOR | MAJOR (must implement it) |
| New optional helper in core | MINOR | No change required |
| Bug fix in a provider only | No change | PATCH on that provider |
| New provider package added | No change | New package at 0.1.0 |

## Release Checklist

Follow these steps every time you publish a new version.

### 1. Decide the version bump

Review commits since the last tag. Use the rules above to decide MAJOR,
MINOR, or PATCH. When in doubt, ask: "Does any consumer need to change their
code to stay compatible?" Yes → MAJOR. No → MINOR or PATCH.

### 2. Update the version in the package file

**TypeScript** — edit `package.json`:
```json
{
  "version": "0.2.0"
}
```

**Python** — edit `pyproject.toml`:
```toml
[project]
version = "0.2.0"
```

Update all packages being released. If core changes, update core first, then
update the `dependencies` field in each provider to reference the new core version.

### 3. Update CHANGELOG.md

Add an entry at the top of `CHANGELOG.md` following the format below.
Never edit past entries — only prepend new ones.

```markdown
## [0.2.0] — 2026-05-01

### Added
- `IRepository.findAll()` optional method on core interface

### Changed
- `loadPostgresConfig()` now also reads `DB_SSL` env var

### Fixed
- Connection not released on rollback failure in MongoUnitOfWork
```

### 4. Commit the version bump

```bash
git add .
git commit -m "chore: release v0.2.0"
```

### 5. Tag the release

```bash
git tag v0.2.0
git push origin main --tags
```

Use a single tag for a coordinated release of all packages. If only one
package changed, tag it specifically:

```bash
git tag data-layer-postgres-v0.2.1
```

### 6. Publish to internal registries

**TypeScript packages** — publish to internal npm registry:
```bash
# Set registry to your internal npm (Verdaccio, Artifactory, etc.)
npm publish --registry https://npm.your-company.internal

# Or if using a scoped registry in .npmrc:
npm publish
```

**Python packages** — build and publish to internal PyPI (Nexus, Artifactory, etc.):
```bash
# Build the wheel
pip install build
python -m build

# Upload to internal PyPI
pip install twine
twine upload --repository internal dist/*
```

### 7. Notify consumers

Post in the team channel with:
- Version number
- Whether it is MAJOR / MINOR / PATCH
- One-line summary of what changed
- Link to the CHANGELOG entry
- If MAJOR: migration steps required

Example message:
> **@company/data-layer v0.2.0 released (MINOR)**
> Added optional `findAll()` to IRepository. No changes required to existing
> implementations. See CHANGELOG for details.

## Consumer pinning

Services that use the data layer should pin to a specific version range,
not `latest`.

**TypeScript** — `package.json`:
```json
{
  "dependencies": {
    "@company/data-layer": "^0.2.0",
    "@company/data-layer-postgres": "^0.2.0"
  }
}
```

`^` allows MINOR and PATCH updates but never MAJOR — this is the recommended
default. Pin to an exact version (`"0.2.0"`) only for services that need
strict reproducibility.

**Python** — `pyproject.toml`:
```toml
dependencies = [
    "company-data-layer>=0.2.0,<1.0.0",
    "company-data-layer-postgres>=0.2.0,<1.0.0",
]
```

## Setting up internal registries

### npm — Verdaccio (self-hosted)

```bash
# Install
npm install -g verdaccio

# Start
verdaccio

# Configure all packages to use it
# Add to .npmrc at project root:
@company:registry=http://localhost:4873
```

### npm — Artifactory / GitHub Packages

Add to `.npmrc`:
```
@company:registry=https://npm.pkg.github.com
//npm.pkg.github.com/:_authToken=${NPM_TOKEN}
```

### Python — Nexus / Artifactory

Add to `pip.conf` or use `--index-url`:
```bash
pip install company-data-layer \
  --index-url https://pypi.your-company.internal/simple/
```

Or in `pyproject.toml` (pip 23+):
```toml
[tool.pip]
index-url = "https://pypi.your-company.internal/simple/"
```

## What must never be committed

These files must stay out of git at all times:

| File | Why |
|------|-----|
| `.env` | Contains real credentials |
| `*.egg-info/` | Build artifact |
| `dist/` | Build artifact |
| `node_modules/` | Installed packages |
| `.venv/` | Python virtual environment |
| `npm-debug.log` | Contains local paths |
| Registry auth tokens | Never hardcode — use env vars or CI secrets |

These are already covered by the root `.gitignore`. Do not bypass it.

## First release (current state)

All packages are currently at `0.1.0` and referenced locally via `file:`
paths. Before publishing to a registry for the first time:

1. Confirm package names match internal naming conventions
2. Verify `exports` fields in `package.json` are correct
3. Run `npm run type-check` in all TypeScript packages — must pass with zero errors
4. Run `python -m mypy` in all Python packages — must pass
5. Run `npm run test:unit` in both TypeScript providers — must pass
6. Run `python scripts/test_unit.py` in both Python providers — must pass
7. Follow the release checklist above starting from step 3
